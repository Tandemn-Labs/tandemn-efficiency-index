# Tandemn Efficiency Index

Tandemn Efficiency Index (TEI) is a Kubernetes-native observability and benchmarking foundation for
GPU inference workloads. It discovers how a model-serving job is configured, attributes GPU and
inference telemetry to the worker Pods that produced it, and maintains a rolling per-job record that
can be inspected through a local dashboard and JSON API.

The project is designed to move beyond a cluster-wide utilization chart. Its data model connects:

- The deployed model and serving-engine configuration.
- Parallelism, batching, KV-cache, scheduling, and placement settings.
- Kubernetes worker identity and runtime role.
- Physical GPU metrics from NVIDIA DCGM.
- Worker-scoped vLLM latency, throughput, queue, token, and KV-cache signals.

This joined record is the input required for future TEI recommendations such as identifying
underutilized hardware, memory pressure, poor workload placement, inefficient parallelism, and
prefill/decode imbalance.

## Current project status

TEI is currently an MVP intended for Tandemn developers, infrastructure teams, and early design
partners validating NVIDIA GPU inference workloads on Kubernetes. The collection path, normalized
models, rolling observer, API, dashboard, and deterministic smoke-test environment are implemented.

The repository is not yet a finished production distribution. It does not currently include:

- The hosted Tandemn account, API-key, upload, or report workflow.
- Koi simulation, efficiency scoring, or automated recommendations.
- Cost-per-token, SLO-margin, canonical chain assignment, or resource-map pricing.
- TLS termination, SSO, or multi-user authorization. A single optional local bearer token is
  supported for API access.

The current dashboard reports observed evidence and directional benchmark bands. It does not claim
to calculate a final Tandemn Efficiency Index score.

## Who TEI is for today

The current implementation is useful for teams that meet most of the following conditions:

- Run LLM inference on NVIDIA GPUs in Kubernetes.
- Deploy models with NVIDIA Dynamo or Ray Serve.
- Use the bundled NVIDIA dcgm-exporter and Prometheus, or provide compatible existing instances.
- Run vLLM or start SGLang with `--enable-metrics` for inference telemetry.
- Can grant the TEI ServiceAccount read access to workload CRDs, resources, and worker Pods.
- Want to validate workload attribution and the quality of collected telemetry before running
  higher-level optimization analysis.

TEI is not currently a general-purpose Kubernetes observability replacement. It is narrowly focused
on collecting the workload and performance context needed to explain and improve model-serving
efficiency.

## Supported frameworks and versions

### Workload discovery

| Runtime | Kubernetes resource | API versions | Parsed serving backends |
| --- | --- | --- | --- |
| NVIDIA Dynamo | `DynamoGraphDeployment`, `DynamoGraphDeploymentRequest` | `nvidia.com/v1beta1`, `nvidia.com/v1alpha1` | vLLM, SGLang, TensorRT-LLM |
| Ray Serve | `RayService` | `ray.io/v1`, `ray.io/v1alpha1` | vLLM, SGLang |

Both detectors normalize their resource into the same `Workload` shape. TEI lists the CRD
definitions visible to its ServiceAccount, identifies the supported served versions, and lists all
visible DynamoGraphDeployments and RayServices. Discovery can cover the cluster or an explicit
namespace allowlist.

When a DGD was generated by a DGDR, TEI uses the DGD's `dgdr.nvidia.com/name` and
`dgdr.nvidia.com/namespace` labels to attach the declared workload, SLO, and hardware intent to the
deployed configuration. Direct DGDs remain valid with no declared intent. Ray has no separate DGDR
equivalent; TEI normalizes per-deployment concurrency and autoscaling controls from
`RayService.spec.serveConfigV2` into the same optional intent section.

### Telemetry

| Source | Current support |
| --- | --- |
| NVIDIA DCGM | Raw exported per-GPU series plus required-profile coverage for utilization, memory, power, thermal, clocks, engine activity, PCIe, and XID |
| vLLM and SGLang | Normalized p99 TTFT/TPOT, request and token throughput, running/waiting requests, and KV-cache usage, with raw backend series retained |
| Dynamo | Frontend latency, request/token throughput, queue/load, and pass-through engine metrics |
| Kubernetes | Dynamo frontend/worker and Ray head/worker Pod identity, phase, readiness, restarts, and resource requests/limits |

TEI intentionally queries DCGM plus the `vllm:*`, `sglang:*`, `dynamo_*`, and `ray_serve_*` metric
families for observed Pods. This bounds query and response cardinality while retaining backend raw
series and emitting a stable normalized inference contract.

At startup TEI lists cluster CRDs and caches the supported resource APIs. Every configured discovery
interval it lists only DGD, DGDR, RayService, and relevant worker Pod instances. The observer then:

1. Reconciles workload resources and component Pods every ten seconds and verifies Prometheus can
   answer a lightweight query.
2. Captures the runtime-generated job key: the DGD name for Dynamo or generated RayCluster name for
   Ray Serve.
3. Writes observation sessions, workload revisions, job-key validity, Pod assignments, and
   collection status to PostgreSQL in one transaction. It does not copy Prometheus samples.
4. When a dashboard or API report is requested, runs one bounded raw range query and eight
   normalization queries for that requested time range.
5. Joins every returned Prometheus series to the Pod assignment and runtime job key valid during
   the observation when identity labels permit attribution.
6. Returns a bounded generated report while Prometheus remains the source of truth for time series.

If a worker stops, its historical data remains in its job record but TEI stops querying new
worker-scoped metrics for it. GPUs without a recognized owner remain visible as unattributed
telemetry instead of disappearing from cluster-level capacity accounting.

Each metric series has a stable identity derived from its metric name, scope, and remaining
Prometheus labels. The PostgreSQL schema contains `observations`, `workloads`, immutable
`workload_revisions`, `runtime_job_keys`, `pod_assignments`, and `state_collection_runs`. There are
no metric-sample tables in TEI. The `runtime_job_key` on each Pod and metric scope is the explicit
join back to its Dynamo or Ray workload.

## Prometheus requirements

The Helm chart installs dedicated Prometheus and dcgm-exporter instances by default. Its Prometheus
configuration explicitly discovers DCGM endpoints, Dynamo metrics-enabled Pods, Ray head/worker
metrics ports, and normal annotation-enabled Pod/Service targets. Set `prometheus.enabled=false`
and provide `prometheus.url` to use an existing server. Set `dcgmExporter.enabled=false` when an
existing exporter is already available. TEI depends only on the standard Prometheus HTTP API and
does not require Prometheus Operator.

The default Kubernetes state interval is ten seconds. A lightweight Prometheus readiness query runs
on each state tick; telemetry range queries run only when a report is requested.
`prometheus.stepSeconds` controls range-query resolution and should normally match the scrape
interval.

### DCGM Pod attribution

dcgm-exporter must run with Kubernetes PodResources attribution enabled. The bundled dependency
enables Pod labels and Pod UIDs automatically:

```text
DCGM_EXPORTER_KUBERNETES=true
```

The exporter identifies the Pod that owns each allocated GPU. When Prometheus adds its own scrape
target `pod` label, the GPU owner's label must be preserved as `exported_pod`. TEI treats:

- `pod` as the dcgm-exporter Pod.
- `exported_pod` as the workload Pod that owns the GPU.
- `exported_namespace` as the optional namespace of that owner.

Without `exported_pod`, physical GPU telemetry is still retained, but it may remain unattributed.
The dashboard and API expose attribution reasons so this configuration problem is visible.

### Prometheus metrics

TEI queries all DCGM series plus bounded inference families exported by observed component Pods for
the requested range, rather than copying samples into its database. It derives
`p99_ttft_ms`, `p99_tpot_ms`, request/input/output throughput, running/waiting requests, and
`kv_cache_usage_ratio` with backend-aware PromQL. DCGM series use exporter ownership labels, while
application series use Pod UID or namespace/Pod identity. The resulting scope includes both
`workload_id` and `runtime_job_key`.

Empty results and non-finite Prometheus values are ignored. A failed report query is exposed as
missing telemetry. Topology- and configuration-dependent metrics such as NVLink remain optional.

## Development setup

Requirements:

- Python 3.12 or newer.
- [`uv`](https://docs.astral.sh/uv/).
- Access to a Kubernetes cluster for live integration testing.
- A reachable Prometheus endpoint for live telemetry.

Install the project and development dependencies:

```shell
uv sync --group dev
```

Run validation:

```shell
uv run ruff format --check src tests smoketest
uv run ruff check src tests smoketest
uv run mypy src smoketest
uv run pytest
uv run pytest smoketest
```

Integration tests must use local containers or test clusters, never real cloud accounts.

## Running the local dashboard with mock data

The repository includes a removable smoke-test environment representing a customer running Qwen,
GLM, and DeepSeek Dynamo jobs. It contains 23 minutes and 17 seconds of deterministic DCGM history,
complete, partial, missing, and unattributed telemetry cases.

```shell
uv run python -m smoketest.run_dashboard
```

Open `http://127.0.0.1:8000` and stop the server with `Ctrl+C`.

The complete smoke-test environment lives under `smoketest/`. Production code does not import it,
so it can be removed as one directory before a production release.

## Running against a cluster

Build the chart dependencies when installing from a source checkout:

```shell
helm dependency build ./helm/tei
```

Build and publish the TEI container, then install the chart. Prometheus, dcgm-exporter, and a pinned
PostgreSQL dependency are included automatically:

```shell
helm install tei ./helm/tei \
  --namespace tei-system \
  --create-namespace \
  --set image.repository=example.com/tandemn-efficiency-index \
  --set image.tag=0.2.0
```

TEI reconciles Kubernetes state every ten seconds and creates bounded 24-hour observations in
PostgreSQL. The bundled Prometheus uses a ten-second scrape interval, a 24-hour retention period,
and a 20 GiB persistent volume. The bundled database uses an 8 GiB persistent volume.

To use an external PostgreSQL database, create a Secret containing its DSN and disable the bundled
dependency. The database role must be able to create and update TEI tables, indexes, and sequences:

```shell
kubectl -n tei-system create secret generic tei-external-postgresql \
  --from-literal=dsn='postgresql://tei:password@postgresql.database.svc:5432/tei'
helm upgrade --install tei ./helm/tei \
  --namespace tei-system \
  --set postgresql.enabled=false \
  --set database.existingSecret.name=tei-external-postgresql
```

To use a Prometheus server that is already running in the cluster:

```shell
helm install tei ./helm/tei \
  --namespace tei-system \
  --create-namespace \
  --set prometheus.enabled=false \
  --set prometheus.url=http://prometheus-operated.monitoring.svc:9090
```

In external mode Helm does not install, modify, or remove Prometheus. The supplied server must
already scrape DCGM, Dynamo frontend/worker, and Ray head/worker targets. Missing source or
normalized signals are reported explicitly in snapshot coverage.

The default `discovery.mode=cluster` creates a TEI ServiceAccount, lists cluster CRD definitions,
and grants cluster-wide read access to DynamoGraphDeployments, DynamoGraphDeploymentRequests,
RayServices, and Pods. The Helm installer must be allowed to create ClusterRoles and
ClusterRoleBindings.

To restrict workload and Pod access to specific namespaces while retaining read-only CRD discovery:

```shell
helm install tei ./helm/tei \
  --namespace tei-system \
  --create-namespace \
  --set discovery.mode=namespaces \
  --set 'discovery.namespaces={inference,research}'
```

In namespace mode the chart binds workload and Pod permissions only in the listed namespaces. CRD
definitions are cluster-scoped, so the ServiceAccount still receives cluster-wide read-only access
to `customresourcedefinitions`. Set `serviceAccount.create=false` and provide
`serviceAccount.name` when a platform administrator supplies RBAC separately.

## Snapshot API

The dashboard server exposes:

```text
GET /api/v1/snapshot
GET /api/v1/status
GET /healthz
GET /readyz
```

Query parameters:

- `window_seconds`: window ending at the record's latest update. Use `0` for all retained data;
  non-zero values are capped at 86,400 seconds.
- `max_points`: maximum serialized points per series, from 2 through 2,000.

Set `auth.bearerTokenSecret.name` and `.key` to protect `/api/*` with a bearer token. The dashboard
prompts once and keeps the token in browser session storage. `/healthz` and `/readyz` remain
unauthenticated for Kubernetes probes; TLS and multi-user policy should be supplied by an ingress or
service mesh.

The response contains:

- Cluster, worker, GPU, metric, and series counts.
- Normalized workload configuration and component settings.
- Attributed worker identities and Dynamo/Ray runtime job keys.
- Job-scoped GPU and inference time series.
- Per-GPU metric coverage.
- Unattributed telemetry and attribution failure reasons.
- Metrics unavailable during the most recent collection tick.

## Privacy and data egress

The current implementation stores workload characteristics and Pod ownership in PostgreSQL and
reads telemetry from the configured Prometheus service. It serves generated reports locally and
does not upload cluster data, call Tandemn-hosted services, or invoke an LLM. Those workflows will
require explicit product and security design before they are added.
